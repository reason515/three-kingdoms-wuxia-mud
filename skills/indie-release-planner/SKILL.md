# Indie Release Planner

## Purpose

Keep the project scoped so it can first prove that consequential choices, historical mystery and multi-run discovery are compelling, then become a credible mobile or desktop indie release.

## Release principles

- Treat mobile as the primary interaction target and Web as the fastest prototype channel.
- Validate immediate replay intent before building formal leaderboards or a large content library.
- Measure whether alternate runs reveal new facts and strategies, not only completion rate.
- Keep save data portable across future Web, mobile and desktop builds.
- Avoid live-service obligations until the finite scenario loop is proven.
- Plan challenge integrity and deterministic seeds before public competition.

## Suggested path

1. **Paper/data prototype**: 8–12 consequential choices, one ending set, rapid internal testing.
2. **Internal mobile Web slice**: one identity-complete route plus at least one meaningfully different replay route.
3. **Complete vertical slice**: three identities, evidence chronicle, character fate summary and six or more endings.
4. **Public Web demo**: collect replay, comprehension and readability data.
5. **Challenge prototype**: fixed scenario seed, local scoring and anti-exploit review.
6. **Platform decision**: evaluate TapTap/mobile packaging first, then Steam/desktop fit based on audience response.

## Core validation metrics

- percentage of players who start another run soon after an ending;
- time from ending to restart;
- identity and route diversity;
- choice distribution and dominant-choice rate;
- percentage who discover genuinely new evidence on later runs;
- character-fate goals players voluntarily pursue;
- abandonment points and reading pace;
- player ability to explain the cost and consequence of major choices.

Do not use a single score as proof of replayability.

## Leaderboard gate

Build a formal leaderboard only after:

- the core scenario is worth replaying without it;
- scoring supports multiple play styles;
- challenge setup is deterministic and auditable;
- save editing, restart abuse and paid advantages have clear policies;
- server and moderation costs are acceptable.

## Required output

When planning releases, include:

1. Current project stage.
2. Next hypothesis to validate.
3. Smallest milestone and feature scope.
4. Mobile, Web and desktop implications.
5. Playtest audience and metrics.
6. Risks and mitigations.
7. Explicit go/no-go criteria.
8. Concrete next actions.

## Avoid

- Designing for every platform at once.
- Building rankings before the replay loop works.
- Premature monetization that damages single-player narrative choices.
- Assuming Web, Steam and mobile have identical UX expectations.
- Promising frequent content seasons before authoring throughput is known.
