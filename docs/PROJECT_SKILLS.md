# 2. 项目专属 Skills

## 2.1 `historical-wuxia-worldbuilder`

用途：将真实三国人物、地点、势力和历史事件转化为“历史夹缝中的江湖故事”。

重点处理：

- 区分不可轻易改变的历史锚点与玩家可介入的史书留白；
- 设计朝廷、军阀、门派、商旅、暗探和地方势力的联系；
- 让历史人物保持政治和时代逻辑，不统一武侠化；
- 让虚构内容解释过程、动机和局部命运，而不是随意推翻史实。

## 2.2 `mud-system-designer`

用途：保留文字 MUD 的规则感、空间感和系统联动。

重点处理：

- 地点、人物、物品、事件和持续世界状态；
- 2～5 个上下文行动与规则执行；
- 时间、伤势、疑迹、人情和情报等稀缺资源；
- 单局、结算、解锁和重开循环；
- 以简洁文字反馈选择后果。

## 2.3 `narrative-quest-writer`

用途：编写历史篇章、人物弧光、多视角真相和不可兼得的选择。

重点处理：

- 史实与虚构的边界；
- 情报真假、身份怀疑和阵营冲突；
- 每个选项的机会成本及延迟后果；
- 分支后收束的可维护剧情结构；
- 主角结局、人物命运图谱与江湖评语。

## 2.4 `combat-and-progression-balancer`

用途：设计服务于叙事的能力构筑和情境化战斗。

重点处理：

- 武、谋、义、势四项核心能力；
- 剑术、轻功、医术、辩才、谍术、兵法等专长；
- 3～6 轮的战斗决策；
- 以解锁行动方式代替单纯数值膨胀；
- 防止永久成长让旧篇章失去挑战。

## 2.5 `game-content-data-architect`

用途：设计可扩展、可校验的内容数据结构。

重点处理：

- 历史篇章、事件节点、选项、条件和状态变更；
- NPC、关系、物品、能力、专长和史证；
- 结局、称号、挑战与统一事件种子；
- 分支可达性、死路和互斥条件检查；
- 文本 key、多语言和版本迁移。

## 2.6 `responsive-text-game-ui-designer`

用途：设计移动端优先、桌面端扩展的文字游戏界面。

重点处理：

- 移动端单列场景文本与底部固定行动区；
- 行动标题、条件、风险和锁定状态；
- 人物、武学、行囊、关系和史卷的单层抽屉；
- 长文本阅读、字号、行距和触控区域；
- 桌面端侧栏与更完整的信息布局。

## 2.7 `localization-i18n-planner`

用途：规划中文原文、英文翻译、专有名词和文本结构。

重点处理：

- 人名、地名、官职、武功和历史术语；
- 选项标题与风险提示的长度；
- UI、剧情、系统反馈和史卷文本分离；
- 文本 key 命名与变量插值。

## 2.8 `indie-release-planner`

用途：规划 Web 验证、移动端发行及未来 Steam / TapTap 路径。

重点处理：

- Demo 范围和里程碑；
- 本地存档、云存档、成就和排行榜；
- 每日或每周统一历史挑战；
- 商店素材、测试计划和合规风险。

# 3. 推荐的最小闭环

第一阶段不制作大型开放世界，先完成《长安残照》垂直切片：

1. **一个历史篇章**：初平三年长安乱局，固定历史大势与可介入的局部命运。
2. **三种初始身份**：提供不同信息、关系和行动方式。
3. **一套选择闭环**：每次从 2～5 个行动中取舍，持续改变资源、关系和疑迹。
4. **一套能力构筑**：核心能力与少量专长直接解锁新选项。
5. **一次情境化战斗**：以文字和 3～6 轮选择验证 MUD 式交锋表现。
6. **多层结算**：主角结局、人物命运、江湖评语和新史证。
7. **多周目验证**：换身份重开后能够看到新的事实，而不只是不同文本。
8. **移动端 Web 原型**：主流程不调用键盘，桌面和手机均可游玩。
9. **数据驱动**：事件、选项、条件、状态变化和文本尽量由数据维护。

# 5. 下一步建议

核心定位已经明确，下一步应优先编写：

1. `docs/STORY_BIBLE.md`：定义史实边界、虚构原则、人物模板和《长安残照》剧情骨架；
2. `docs/SYSTEM_DESIGN.md`：定义单局状态、选项规则、资源、成长、结算和重开；
3. `docs/CONTENT_SCHEMA.md`：定义事件节点、条件、效果、史证和结局的数据结构；
4. 一个只包含 8～12 次选择的纸面原型，用于验证取舍和重玩动力。
