# 1. 文档目标

本文把剧情文档中已经出现的规则统一为一套可执行、可测试、可扩展的游戏机制。

设计目标不是提前制作完整大型 MUD，而是确定一组稳定的规则内核，使《长安残照》、后续历史篇章和未来平台扩展能够共享同一种状态、条件、效果与结算方式。

核心原则：

> MVP 内容可以少，但规则不能靠每个剧情节点临时解释。

# 2. 机制设计原则

## 2.1 确定性优先

MVP 的关键剧情不使用随机成功率。结果由以下内容共同决定：

- 当前状态；
- 已掌握武学；
- 装备和物品；
- 人物关系；
- 前序选择；
- 当前选项明确声明的代价。

例如：

```text
已掌握“踏墙换位”且未穿札甲
→ 成功越过屋脊

未掌握，但无伤且身法条件满足
→ 可以勉强越过，同时受到轻伤

重伤或穿札甲
→ 选项锁定
```

未来可以为挑战模式、探索事件增加可选随机模块，但不能让随机结果破坏主线因果或排行榜公平。

## 2.2 离散状态优先

玩家面对的是选择，不是计算器。MVP 优先使用少量等级、标签和枚举，不使用复杂连续公式。

- 伤势使用五级状态；
- 风声使用四级状态；
- 人物态度使用五级状态；
- 能力使用 0～3 级；
- 结局由条件和优先级判定。

内部可以使用整数存储，但界面优先显示有意义的词语。

## 2.3 规则与文本分离

文字负责氛围和解释，规则负责唯一真实状态。

不能通过“文本里说玩家受伤了，但状态没有改变”来推进剧情。所有会影响后续的结果必须写入状态、关系、物品、标记或调度队列。

## 2.4 内容数据驱动

事件、选项、条件、效果、武学、装备、人物和结局尽量由结构化内容描述。引擎只实现通用操作，不为单个剧情写专用分支代码。

## 2.5 扩展不等于预先实现

为未来保留稳定接口，但 MVP 不实现：

- 大规模随机掉落；
- 完整经济系统；
- 多人在线；
- 程序生成剧情；
- 数十层武学境界；
- 正式排行榜服务端。

# 3. 规则分层

游戏规则分为五层：

```text
玩家表达层
→ 事件与选项层
→ 条件与效果层
→ 单局状态层
→ 周目档案与跨局状态层
```

## 3.1 玩家表达层

玩家看到：

- 场景文字；
- 当前关键状态；
- 2～5 个行动；
- 行动要求；
- 主要风险；
- 选择后的叙事与状态反馈。

玩家不需要理解内部 ID、条件树和结局优先级。

## 3.2 事件与选项层

事件负责定义当前情境，选项负责表达玩家可以采取的行动。

一个事件包含：

- 触发条件；
- 地点和时间阶段；
- 出场人物；
- 场景文本；
- 选项；
- 无可用选项时的兜底；
- 默认后续事件或结局检查。

## 3.3 条件与效果层

条件只读取状态，效果只修改状态。条件和效果都使用有限的通用操作符，不嵌入任意脚本。

## 3.4 单局状态层

保存本轮经历中的一切事实，包括人物状态、物品、关系、地点、已知信息和行动日志。

## 3.5 跨局状态层

保存人物武谱、江湖见闻、结局、挑战与设置，不把上一局的全部战力直接带入下一局。

# 4. 核心状态模型

## 4.1 能力

MVP 使用三项可比较能力：

| 能力 | 取值 | 作用 |
|---|---:|---|
| 武 | 0～3 | 正面交锋、护卫、威慑、强行突破 |
| 谋 | 0～3 | 观察、判断、布置、识破和欺骗 |
| 势 | 0～3 | 身份、人脉、军中或府中影响力 |

等级含义：

- 0：不具备；
- 1：略懂或身份有限；
- 2：可靠专长；
- 3：篇章内的突出能力。

## 4.2 “义”不作为数值能力

“义”如果设计为统一加减的数值，容易退化为善恶条。机制上改为**行止记录**：

- `守诺`；
- `背诺`；
- `护弱`；
- `弃人`；
- `不杀`；
- `滥杀`；
- `代人受过`；
- `以自由换人`；
- `保存真相`；
- `毁证救人`。

NPC 根据自己的价值观解释这些行为。守诺可能得到赵衡认可，却让阿萝敌视；背诺也可能是阻止不义命令。

界面可以显示“行止：未定”或本局已经形成的标签，不显示“义 +1”。

## 4.3 局内资源

| 资源 | MVP范围 | 规则 |
|---|---:|---|
| 内息 | 0～3，少数情况临时提高上限 | 使用进阶招式、强行压伤或连续行动 |
| 余时 | 0～3 | 表示历史节点到来前可自由支配的时间段 |
| 人情 | 0～2，通常初始 1 | 请求指点、报信、治疗、掩护或接应 |

资源不得出现负数。支付不足时选项应锁定或提供有额外代价的替代结果。

## 4.4 伤势

| 内部值 | 玩家显示 | 默认影响 |
|---:|---|---|
| 0 | 无伤 | 无限制 |
| 1 | 轻伤 | 部分连续强攻代价增加 |
| 2 | 重伤 | 轻身、强攻和长时间护卫受限 |
| 3 | 濒危 | 下一次重创通常导致死亡 |
| 4 | 死亡 | 终止常规行动，进入结算 |

伤势只能通过明确治疗、休息、药物或特殊武学下降。普通场景切换不会自动恢复。

## 4.5 风声

| 内部值 | 玩家显示 | 默认影响 |
|---:|---|---|
| 0 | 平静 | 无人专门关注玩家 |
| 1 | 起疑 | 局部人物注意到异常 |
| 2 | 暴露 | 身份或行动已经被部分敌人确认 |
| 3 | 缉拿 | 公开搜捕，盘查和追兵升级 |

风声是外界警觉，不是潜行技能，也不等于被捕。

风声通常只升不降；换装、制造假线索或转移身份可以降低一级，但必须有具体场景依据。

## 4.6 生死与自由

### 生死状态

```text
alive / dead / missing
生还 / 死亡 / 生死不明
```

### 自由状态

```text
free / captured / imprisoned / escaped / coerced
自由 / 落网 / 收押 / 脱逃 / 被迫效力
```

状态约束：

- 风声达到缉拿且自由状态仍为自由时，玩家处境显示为“被追捕”，但内部自由状态不变；
- 只有事件明确控制住玩家，才变为落网；
- 落网后仍有选择时，可以进入一个短续段；
- 失去后续行动能力时，变为收押并进入结局检查；
- 死亡后自由状态不再参与行动，但可以保留作结算记录。

## 4.7 位置与场景状态

单局至少记录：

- 当前地点；
- 当前事件；
- 当前历史阶段；
- 当前战斗轮次；
- 与玩家同行的人；
- 当前保护目标；
- 当前可用退路和环境标签。

环境标签示例：

```text
狭窄 / 屋脊 / 火场 / 高处 / 人群 / 军阵 / 可藏身 / 可骑乘
```

武学和装备通过环境标签解锁行动，避免为每个地点硬编码规则。

# 5. 人物关系与命运

## 5.1 关系模型

每组人物关系由两部分构成：

1. **态度值**：内部 -2～2；
2. **关系标签**：描述不可被一个数字替代的事实。

| 内部值 | 玩家显示 |
|---:|---|
| -2 | 敌视 |
| -1 | 疏离 |
| 0 | 中立 |
| 1 | 信任 |
| 2 | 托命 |

关系标签示例：

- 同门；
- 救命之恩；
- 师徒；
- 欠债；
- 承诺；
- 背叛；
- 仇怨；
- 共同秘密。

同门不会因为态度下降而消失，救命之恩也不等于永远信任。

## 5.2 关系反馈

玩家看到语义反馈，不必看到数值：

```text
阿萝对你的戒备稍缓。
赵衡已经不再把你视作同门。
你欠下了许简一份无法当场偿还的人情。
```

开发与测试模式可以显示内部值。

## 5.3 人物命运状态

NPC 命运使用可扩展状态：

```text
安全 / 同行 / 受伤 / 被捕 / 失踪 / 死亡 / 投靠某方 / 成为敌人
```

一个角色可以同时拥有多个不冲突状态，例如“重伤 + 被捕”。最终结算根据状态生成命运文本。

# 6. 事件与选项机制

## 6.1 事件类型

MVP 支持：

- `scene`：普通剧情场景；
- `choice`：主要抉择；
- `combat`：一至多轮交锋；
- `growth`：武学定路、参悟或指点；
- `capture`：落网后的短续段；
- `ending`：结算入口。

未来可以增加旅行、交易、营地、调查和多人事件，但仍复用同一条件与效果系统。

## 6.2 选项可见状态

每个选项只能处于以下状态之一：

- **可用**：可以立即选择；
- **锁定可见**：展示缺少的武学、物品或关系，用于提示重玩可能；
- **隐藏**：角色当前不可能想到，或显示会泄露秘密；
- **已失效**：曾经可用，但因时间、人物或物品变化关闭。

普通场景显示 2～3 个核心选项，复杂抉择 3～4 个，终局每轮最多 5 个。

## 6.3 可见信息

一个选项可以显示：

- 行动标题；
- 主要意图；
- 已知条件；
- 明确代价；
- 风险类型；
- 锁定原因。

不显示角色不可能知道的精确后果。

风险等级建议：

```text
无明显风险 / 有代价 / 高风险 / 致命风险
```

## 6.4 选项有效性标准

一个有效选择至少应改变以下一项，优先改变两项：

- 资源；
- 伤势、风声或自由状态；
- 武学成长；
- 物品或装备；
- 人物关系或命运；
- 路线和后续事件；
- 已知信息；
- 延迟后果；
- 行止标签。

只改变一句对白、但不改变状态或认知的内容不应成为独立选项。

# 7. 条件系统

## 7.1 通用条件操作

MVP 只实现有限条件：

- `all`：全部成立；
- `any`：任一成立；
- `not`：取反；
- `eq` / `neq`：相等或不等；
- `gte` / `lte`：等级比较；
- `hasFlag`：拥有标记；
- `hasTag`：拥有标签；
- `hasItem`：持有物品；
- `itemState`：物品状态；
- `relationship`：人物态度或标签；
- `atLocation`：当前位置；
- `knows`：已知信息；
- `techniqueKnown`：已掌握招式。

## 7.2 组合示例

```yaml
all:
  - techniqueKnown: roam.step_wall
  - injury: { lte: 1 }
  - not:
      hasTag: equipment.heavy_armor
  - environmentHas: rooftop
```

## 7.3 扩展规则

未来新增条件时，必须：

- 有稳定名称；
- 有确定输入和输出；
- 可以被验证工具识别；
- 不能直接执行任意代码；
- 不能只服务一个事件而没有复用价值。

# 8. 效果与结算顺序

## 8.1 通用效果操作

MVP 支持：

- `set`：设置状态；
- `inc` / `dec`：有限数值增减；
- `addFlag` / `removeFlag`；
- `addTag` / `removeTag`；
- `giveItem` / `removeItem`；
- `setItemState`；
- `equip` / `unequip`；
- `changeRelationship`；
- `setFate`；
- `learnTechnique`；
- `addInsight`；
- `addKnowledge`；
- `moveTo`；
- `scheduleEvent`；
- `cancelEvent`；
- `emitFeedback`。

## 8.2 固定结算顺序

每次选择按以下顺序执行：

1. 再次验证选项条件；
2. 支付显式资源与物品代价；
3. 更新装备和物品；
4. 更新伤势、内息、余时、风声与自由状态；
5. 更新人物关系和命运；
6. 更新武学、知识、标记和行止；
7. 调度或取消延迟事件；
8. 执行状态规范化；
9. 检查死亡、落网续段和强制结局；
10. 选择下一个事件；
11. 生成叙事反馈和状态摘要；
12. 写入行动日志并自动存档。

## 8.3 状态规范化

用于防止非法状态：

- 内息和资源不得低于 0；
- 伤势达到死亡时，生死状态自动变为死亡；
- 已被没收或损坏的物品不能继续装备；
- 收押状态不能进入普通移动事件；
- 互斥武学路线不能同时设为本局主路线；
- 已死亡人物不能再次作为正常同行者；
- 余时低于 0 时固定为 0，并触发相应历史阶段检查。

# 9. 时间与事件调度

## 9.1 余时不是现实分钟

余时代表玩家在不可逆历史节点到来前还能完成多少件耗时行动。

《长安残照》MVP：

- 3：仍有选择路线的余裕；
- 2：城门开始收紧；
- 1：败兵与百姓涌向城门；
- 0：城乱或城破阶段到来。

## 9.2 时间推进

以下行为通常消耗余时：

- 治疗和休息；
- 参悟秘籍；
- 绕路护送；
- 深入追查；
- 处理需要准备的伪装。

普通阅读、对话和同一事件中的战斗轮次不自动消耗余时，除非选项明确声明。

## 9.3 调度事件

事件可以被安排在：

- 下一个节点；
- 余时降到指定值；
- 风声达到指定级别；
- 某人物到达地点；
- 某历史阶段开始；
- 指定延迟选择次数之后。

同一时刻多个事件可触发时，按优先级与确定性排序，不随机抽取主线事件。

# 10. 战斗机制

## 10.1 设计目标

战斗不是独立数值小游戏，而是高压力的连续决策。每轮必须改变距离、位置、威胁、目标或资源中的至少一项。

## 10.2 战斗状态

一场战斗至少记录：

- 轮次；
- 距离：远、中、近、贴身；
- 玩家位置与环境标签；
- 当前主要威胁；
- 保护目标；
- 敌人状态；
- 可用退路；
- 是否允许交涉、撤退或投降。

敌人状态不使用大型 HP 条：

```text
稳势 / 受制 / 受伤 / 失衡 / 被卸兵 / 败退 / 被擒 / 死亡
```

## 10.3 每轮结构

```text
描述环境与距离
→ 预告敌方威胁
→ 展示2～5个行动
→ 结算行动与敌方回应
→ 更新状态
→ 判断继续、脱离、落网或结束
```

## 10.4 战斗行动类别

- 抢攻；
- 守势；
- 试探；
- 使用招式；
- 使用装备；
- 借助环境；
- 保护同伴；
- 威慑或欺骗；
- 撤退；
- 投降或交换。

不提供永远存在的“普通攻击”按钮。普通攻击只有在当前局势下具有明确意义时出现。

## 10.5 胜负不是二元结果

战斗结算至少考虑：

- 是否击败、摆脱或拖住敌人；
- 是否保护目标；
- 是否负伤或死亡；
- 是否消耗内息和装备；
- 是否提高风声；
- 是否错过时间；
- 是否被捕；
- 是否杀人；
- 对手是否成为后续人物。

## 10.6 撤退与投降

撤退和投降是正式战术选项：

- 撤退通常保命，但放弃地点、目标、物品或时间；
- 投降至少失去两项重要资源或行动权；
- 主动受缚可以换取他人安全；
- 落网后最多追加一个短续段，避免支线无限膨胀。

# 11. 伤势、死亡与失败前进

## 11.1 伤势推进

一次结果默认最多提高一级伤势。只有明确标记为致命风险的行为、坠落、火场、弓弩近射或主动舍命可以跨级。

## 11.2 死亡条件

死亡来自：

- 濒危后再次受到重创；
- 玩家选择明确的舍命行动；
- 玩家已知的致命环境后果；
- 特定处决结局。

不能使用隐藏随机数令健康角色突然死亡。

## 11.3 死亡不是空结算

死亡后仍结算：

- 主角完成的事情；
- 获救或受害人物；
- 武学与装备去向；
- 史证和传承；
- 江湖评语；
- 跨周目解锁。

## 11.4 非死亡失败

失败可以表现为：

- 目标逃脱；
- 人物受伤或不再信任；
- 装备损坏；
- 风声上升；
- 错过事件；
- 被迫改变路线；
- 落网或被迫效力。

除非状态已经没有有意义的行动，失败不应立即结束周目。

# 12. 武学与成长机制

## 12.1 武学结构

```text
武学体系
├─ 根基
├─ 路线
├─ 招式
└─ 特殊用法
```

示例：

```text
太原短剑
├─ 短剑根基
├─ 守剑
│  ├─ 横剑截流
│  └─ 借力退锋
└─ 游剑
   ├─ 踏墙换位
   └─ 引锋离路
```

## 12.2 招式状态

| 状态 | 含义 |
|---|---|
| 未知 | 角色不知道存在 |
| 见闻 | 知道招式或传承线索 |
| 可参悟 | 满足根基和剧情条件 |
| 已掌握 | 可以在符合环境的选项中使用 |
| 精通 | 未来扩展状态，MVP 不实现数值成长 |

## 12.3 领悟

领悟不是通用经验值。每条路线独立记录少量主题行为：

- 守剑领悟来自护人、守路和以防守争取时间；
- 游剑领悟来自换位、引敌和放弃正面缠斗；
- 破阵领悟来自突破与压制；
- 护军领悟来自维持队形与替人承伤。

同一低风险事件不能重复获得领悟。

## 12.4 定路与成招

MVP 每局：

- 最多选择一条主武学路线；
- 最多掌握一至两个进阶招式；
- 至少一个招式必须在后续事件中有实际用途；
- 放弃练功必须能够换取时间、人物或装备价值。

## 12.5 未来扩展

后续篇章可以加入：

- 同一人物跨篇章的武学延续；
- 师门和传承关系；
- 招式组合；
- 克制与拆招；
- 兵器适配；
- 伤病造成的武学变体。

扩展继续使用“条件解锁行动”，不转向无限数值膨胀。

# 13. 秘籍机制

## 13.1 秘籍与招式分离

秘籍是物品，招式是人物知识。持有秘籍不等于掌握招式。

```text
发现 → 辨认 → 满足根基 → 支付代价参悟 → 实战掌握
```

## 13.2 秘籍状态

- 完整；
- 残缺；
- 烧损；
- 被批注；
- 真伪不明；
- 已托付；
- 已毁。

## 13.3 秘籍用途

秘籍可以：

- 学习；
- 交换；
- 赠予；
- 归还师门；
- 引火或毁证；
- 留给后人；
- 作为人物关系凭证。

## 13.4 未来扩展

未来可以支持多页残卷、不同批注者和互相矛盾的版本，但应把“版本”作为数据，不为每本秘籍创建新代码。

# 14. 装备与物品机制

## 14.1 槽位

MVP：

- 兵器 × 1；
- 衣甲 × 1；
- 信物 × 1；
- 消耗品 × 2；
- 剧情物品不占普通槽位。

未来可以增加副手、坐骑和工具，但不改变物品的条件与效果接口。

## 14.2 物品状态

通用状态包括：

- 完好；
- 受损；
- 损毁；
- 隐藏；
- 被没收；
- 已托付；
- 遗失。

## 14.3 装备设计标准

每件重要装备必须：

1. 解锁至少一个行动；
2. 带来一个明确优势；
3. 带来一个限制或暴露风险；
4. 在至少两个事件中有用途；
5. 允许被放弃、损坏、赠予或带入结局。

不满足以上要求的物品应合并为普通叙事道具。

## 14.4 换装与没收

- 换装发生在事件选项中，不提供无时间成本的全局换装；
- 装备新物品时必须处理原槽位物品去向；
- 被捕时按场景规则没收可见兵器、信物和秘籍；
- 隐藏物品需要提前设置，不能落网后临时声明；
- 被没收物品仍存在于世界中，可被其他人物或周目发现。

## 14.5 未来扩展

不改变核心接口即可增加：

- 坐骑；
- 毒药与解药；
- 多用途工具；
- 兵器传承；
- 简单耐久；
- 固定词条的名器。

随机属性、强化与套装必须在核心循环验证后单独评估。

# 15. 知识、传闻与史证

## 15.1 知识不是统一情报值

每条知识记录：

- 主题；
- 内容；
- 来源；
- 可信度；
- 是否亲历；
- 支持或冲突的其他说法；
- 是否跨周目保留。

## 15.2 可信度

```text
传闻 / 有来源 / 亲历 / 多方互证 / 已确认
```

可信度影响玩家提示和史卷展示，不自动替玩家判断道德立场。

## 15.3 跨人物见闻

其他人物周目提供的见闻只能：

- 显示新选项；
- 揭示弱点或路线；
- 提高判断依据；
- 解锁新的起始配置。

见闻不能直接替代当前人物的武学、装备和关系条件。

# 16. 结局系统

## 16.1 结局不是单一枚举

完整结算由多个维度组成：

```text
结局族
+ 生死状态
+ 自由状态
+ 关键人物命运
+ 武学传承
+ 装备去向
+ 行止标签
+ 江湖评语
```

## 16.2 结局判定优先级

1. 规范化所有状态；
2. 确定生死与自由状态；
3. 计算关键人物命运；
4. 按优先级匹配主结局族；
5. 追加武学、行为和挑战标签；
6. 计算秘籍与装备去向；
7. 生成江湖评语；
8. 写入跨局解锁。

## 16.3 主结局冲突

每个主结局具有：

- `priority`；
- 必需条件；
- 排除条件；
- 必付代价；
- 兜底文本。

同一局只能显示一个主结局族，但可以同时具有多个维度和标签。

## 16.4 不使用统一道德分

结局不计算“善良总分”。评价由具体行为、人物立场和结果生成。

例如：

- 赵衡赞赏守令；
- 阿萝记住护人；
- 王允府记录背叛；
- 江湖传闻称其不杀而胜。

这些评价可以同时成立。

# 17. 单局流程与事件解析

## 17.1 新开一局

1. 选择历史篇章；
2. 选择已解锁人物；
3. 选择允许的起始配置；
4. 创建单局状态；
5. 应用历史挑战修正；
6. 进入起始事件。

## 17.2 事件解析器

当一个事件结束时：

1. 收集已调度事件；
2. 过滤不满足条件的事件；
3. 按强制历史事件、人物事件、地点事件、普通事件排序；
4. 同优先级按稳定 ID 排序；
5. 选择第一个事件；
6. 若无事件，执行兜底或结局检查。

MVP 不随机抽取主线事件。未来随机事件使用独立事件池和固定种子。

## 17.3 历史锚点

历史锚点由场景阶段驱动，不允许普通选项取消。

玩家可以改变：

- 到达锚点时的状态；
- 谁在场；
- 谁生还；
- 局部原因和见证；
- 史卷留下什么。

玩家不能通过通用效果删除“长安失守”等锚点。

# 18. 多周目与长期成长

## 18.1 三层存档

### 单局状态

- 当前事件和地点；
- 状态、资源、关系、物品；
- 调度事件；
- 行动日志。

### 人物武谱

- 已发现武学路线；
- 曾掌握招式；
- 秘籍完整度；
- 装备经历；
- 人物结局与标签。

### 江湖史卷

- 历史锚点；
- 人物命运版本；
- 见闻与史证；
- 跨人物线索；
- 已解锁身份和挑战。

## 18.2 永久成长边界

永久成长优先解锁：

- 新人物；
- 新起始配置；
- 新武学方向；
- 新事件和见闻；
- 特殊挑战。

不默认永久增加所有人物的武、谋、势。

## 18.3 未来跨篇章结构

未来可以采用：

- 同一人物继续进入后续历史篇章；
- 同门、后辈或继承者承接武学；
- 某件兵器跨篇章流传；
- 前一篇章的人物命运改变后一篇章事件池。

跨篇章只传递明确的档案数据，不复制整份旧单局状态。

# 19. 存档、回放与挑战

## 19.1 自动存档

每次选择结算后自动保存。存档用于中断后继续，不默认允许撤销已经完成的选择。

普通探索模式未来可以提供辅助性的“重看本幕”，但挑战与排行榜模式必须禁用回退。

## 19.2 行动日志

每局记录：

- 事件 ID；
- 选项 ID；
- 选择前关键状态；
- 执行效果；
- 选择后关键状态；
- 内容版本；
- 挑战种子。

行动日志用于：

- 调试；
- 纸面规则校验；
- 结局回顾；
- 排行榜防作弊；
- 未来的文字战报分享。

## 19.3 版本迁移

存档必须包含 `schemaVersion` 和 `contentVersion`。

- 规则结构变化使用 schema 迁移；
- 文本和数值调整使用内容版本；
- 无法安全迁移的旧单局可以结算为历史记录，但不得损坏跨局档案。

# 20. 未来扩展接口

## 20.1 保持稳定的内核

以下概念从 MVP 起保持稳定：

- 事件；
- 选项；
- 条件；
- 效果；
- 人物状态；
- 关系与标签；
- 物品与装备状态；
- 武学与招式；
- 调度事件；
- 结局维度；
- 单局与跨局存档。

## 20.2 可插拔扩展

未来模块通过通用接口接入：

| 模块 | 接入方式 |
|---|---|
| 随机事件 | 条件化事件池 + 固定种子 |
| 坐骑与旅行 | 新装备槽 + 地点/时间效果 |
| 经济 | 资源类型 + 交易事件，不改核心选择流程 |
| 制作与锻造 | 物品效果与状态转换 |
| 阵营 | 关系模型的组织级实例 |
| 同伴 | 多角色状态与战斗保护目标 |
| 排行榜 | 行动日志、挑战种子与结局指标 |
| 多语言 | 文本 key 与变量，不改变规则数据 |
| AI润色 | 只改表现层，不生成权威状态和结果 |

## 20.3 需要重新立项的扩展

以下功能不能假定可以低成本接入：

- 真正多人在线世界；
- 玩家自由交易经济；
- 实时动作战斗；
- 完全开放式 AI 跑团；
- 大规模程序生成历史剧情。

若未来需要，应单独评估产品定位和架构，而不是污染当前规则内核。

# 21. 内容数据最小形态

以下示例只表达规则结构，完整 Schema 见 `docs/CONTENT_SCHEMA.md`。

```yaml
id: changan.dujian.d09
kind: combat
location: changan.west_alley
stage: escort
textKey: scene.changan.dujian.d09
choices:
  - id: guard_the_alley
    labelKey: choice.changan.dujian.d09.guard
    visibility:
      techniqueKnown: dujian.guard.hold_line
    costs:
      - dec: { path: player.breath, value: 1 }
    effects:
      - inc: { path: player.injury, value: 1, max: 4 }
      - addTag: conduct.protect_weak
      - setFate: { character: civilians, state: protected }
      - addFlag: dujian.held_alley
    next: changan.dujian.d10
```

数据设计要求：

- 选项 ID 稳定；
- 文本通过 key 引用；
- 条件和效果可由工具验证；
- 不把任意代码写进内容；
- 每个效果都能在行动日志中解释。

# 22. 校验规则

内容构建或测试时至少检查：

1. 每个可达非结局事件至少有一个可用选项；
2. 每轮向玩家展示的核心选项不超过 5 个；
3. 所有引用的事件、人物、物品、武学和文本 key 存在；
4. 资源不会低于 0；
5. 死亡、收押和普通行动状态不冲突；
6. 被没收或损坏物品不会继续提供能力；
7. 互斥主武学路线不会同时成立；
8. 每个已学进阶招式至少在后续一个节点有用途；
9. 每件关键装备至少影响两个事件；
10. 每个主结局可达，且同一局的主结局优先级唯一；
11. 历史锚点不会被普通效果取消；
12. 跨周目见闻不会绕过当前人物的能力与物品要求；
13. 落网续段最多一层，不递归生成新的收押支线；
14. 没有无代价的完美路线。

# 23. 杜缄路线规则校验重点

在把 `docs/DU_JIAN_ROUTE.md` 转成数据前，优先验证：

- 余时从 3 开始是否足以支持至少两种合理路线；
- D02、D03、D05 的行为能否稳定解锁守剑或游剑；
- 穿札甲关闭游剑是否有明确提示和补救机会；
- D07 放弃练功换时间是否与学招路线同样有价值；
- D09 学会的招式是否立即产生用途；
- D10 主动受缚是否至少失去两项重要资源；
- D11 濒危、落网和死亡检查是否顺序正确；
- D12 主结局与生死、自由状态能否独立组合；
- 任朔见闻 `zhao_shoulder_known` 是否只提供新方法而非自动胜利。
